# sid_planner — описание программы

## Что делает

Программа автоматически распределяет адреса (SID) между всеми узлами и арендаторами в сети провайдера — так, чтобы каждый адрес был уникальным, все адреса укладывались в выделенный провайдеру IPv6-блок и оставался запас на рост. Если параметры не влезают, программа объясняет почему и предлагает конкретные варианты исправления.

Дополнительно генерирует ULA-адреса для физических стыков коммутатор↔сервер и проверяет что они не пересекаются с глобальным пулом SID.

## Запуск

```bash
python sid_planner.py                   # интерактивный режим
python sid_planner.py --example 1       # малый провайдер
python sid_planner.py --example 2       # средний провайдер
python sid_planner.py --example 3       # пример с ошибкой и советами
python sid_planner.py --example 1 --csv # сохранить результат в CSV
```

## Входные параметры

**Провайдерский IPv6-префикс** — адресный блок который провайдер получил от регистратора (RIPE NCC). Например `2001:db8::/32`. Все генерируемые адреса будут лежать внутри этого блока.

**Число дата-центров (D)** — сколько площадок у провайдера. Влияет на то сколько бит в адресе отводится под идентификатор ЦОД.

**Максимум leaf-узлов в одном ЦОД (L)** — сколько серверных коммутаторов максимум в каждом ЦОД. Каждый leaf получает свой уникальный адресный префикс (локатор).

**Максимум арендаторов на leaf (T)** — сколько независимых виртуальных сетей (VRF) может быть на одном узле. Под каждого арендатора выделяется отдельный SID.

**block_len** — сколько бит в адресе отводится под «координаты» узла (ЦОД + leaf). Рекомендуется 40. Чем больше провайдер, тем больше нужно бит.

**node_len** — сколько бит отводится под локатор узла внутри block. Рекомендуется 24. Оставшиеся биты (128 − block_len − node_len) идут под номер арендатора и тип функции.

**Поведения SID (behaviors)** — какие типы адресов генерировать для каждого арендатора. `dt4` — адрес для IPv4-трафика арендатора, `dt6` — для IPv6-трафика. Если выбрать оба, на каждого арендатора генерируется два SID.

## ULA-адреса для физических стыков

Помимо глобальных SID программа генерирует ULA-адреса (Unique Local Addresses, RFC 4193) для физических стыков коммутатор↔сервер.

**Зачем два типа адресов.** На каждом коммутаторе в реальной инфраструктуре есть два вида адресов:
- **Глобальные** — маршрутизируются по всей сети провайдера, из них берутся SRv6 SID
- **ULA** — используются только на физическом линке между коммутатором и сервером, дальше не маршрутизируются

Программа генерирует обе таблицы и явно проверяет что они не пересекаются. ULA лежат в диапазоне `fd::/8`, глобальные SID — в диапазоне провайдера (например `2001:db8::/32`), поэтому пересечений быть не может по определению RFC 4291 — но проверка всё равно выполняется и отображается в отчёте.

**Дополнительные параметры для ULA:**

**Максимум серверов на leaf** — сколько физических серверов подключено к каждому leaf-коммутатору. Под каждый стык генерируется /127 (два адреса: ::0 на коммутаторе, ::1 на сервере).

**ULA Global ID провайдера** — 40-битный уникальный идентификатор провайдера внутри ULA-пространства. По умолчанию `fd12:3456:789a`. В продакшене выбирается случайно согласно RFC 4193.

## Что получается на выходе

**В консоли:**
- Схема разбиения адресного пространства (сколько бит под что)
- Утилизация пространства и резерв на рост
- Таблица локаторов (первые 6)
- Примеры SID для первых трёх арендаторов
- Таблица ULA-стыков (первые 6)
- Проверка отсутствия конфликтов

**В CSV (флаг `--csv`)** — три файла рядом со скриптом:
- `*_locators.csv` — все локаторы
- `*_sids.csv` — полная таблица SID
- `*_ula_links.csv` — все физические стыки с ULA-адресами

## Зачем это нужно

Когда разворачиваешь SRv6 на инфраструктуре провайдера, нужно раздать уникальные адреса (SID) всем узлам и арендаторам так, чтобы они иерархически укладывались в твой IPv6-блок и не конфликтовали между собой. Вручную это таблица в Excel на тысячи строк с неизбежными ошибками — программа делает это за секунду и сразу проверяет что конфликтов нет.

Стандартная рекомендация для локатора — префикс /64 (block_len + node_len = 64), именно под неё настроены параметры по умолчанию. Оставшиеся 64 бита полностью уходят под Function-часть — номер арендатора и тип функции. При 16-битном поле функции это до 65 535 VRF на один узел, чего с запасом хватает для любого реального провайдера.

## Что делать если не влезает

Если программа выдаёт ошибку — она сама предложит три варианта:
1. Расширить провайдерский блок (запросить у RIPE NCC префикс покороче, например /28 вместо /32)
2. Увеличить block_len
3. Таблицу допустимых комбинаций DC × leaf при текущих ограничениях — можно выбрать подходящую
